# 执行兜底与跨源根因分析

> 对标官方高分攻略中「控制 AI 推理不可预测性」与「从被动响应进化为主动诊断」的要求。本模块在「感知-决策-护栏-执行」主闭环之上，补齐三项**确定性兜底**与**智能诊断**能力：执行轮次熔断（防失控/死循环）、一键回滚账本（变更可逆）、跨源根因分析（指标↔日志关联 + L1–L3 分级）。三者均不破坏既有护栏与评测体系的确定性。

## 1. 为什么补这三项

赛题核心挑战是「把 AI 推理的不可预测性约束在安全护栏内」。除了「执行前」的意图校验，一个可信的运维 Agent 还需要「执行中」的失控兜底与「执行后」的可逆与可解释：

| 阶段 | 风险 | 本模块对策 |
| --- | --- | --- |
| 执行中 | 模型规划出过长指令链 / 自我循环，导致失控 | 最大执行轮次熔断（确定性上限） |
| 执行后 | 变更已落地但需要快速回退 | 一键回滚账本（补偿回放 + 人工指引） |
| 诊断 | 单点感知只看「某项异常」，缺乏跨源因果 | 跨源根因分析（指标↔日志关联 + 分级处置） |

## 2. 执行轮次熔断（确定性上限）

### 2.1 设计

`ExecuteNode` 在执行计划时统计**已真实执行的命令数**（只读 SAFE 与确认放行的 REVIEW 命令计入），一旦达到 `ops.exec.max-steps-per-request` 上限，剩余命令一律停执并标记：

```
[circuit] 已达单次最大执行轮次上限(N)，为保证关键任务确定性、防止失控与死循环，剩余指令暂停执行，请分批处理或人工介入
```

- 默认上限 **20**，远大于任意单条评测场景的命令数，因此 **33/33 确定性回放不受影响**；
- 仅作用于执行层，BLOCK/INJECTION_BLOCKED 仍在更早阶段短路，互不干扰；
- 节点输出新增 `executedCount` / `maxStepsPerRequest` / `cappedCount`（被熔断条数，>0 时出现），可追溯。

### 2.2 配置

| 配置项 | 默认 | 环境变量 |
| --- | --- | --- |
| `ops.exec.max-steps-per-request` | 20 | `OPS_EXEC_MAX_STEPS` |

### 2.3 验证

以 `OPS_EXEC_MAX_STEPS=1` 重启后触发多命令计划，后续命令 `output` 即出现 `[circuit]` 提示；验毕恢复默认即可。

## 3. 一键回滚账本

### 3.1 设计

每次进入执行层的变更命令都会被 `RollbackLedger` 记账，`RollbackAdvisor` 依据命令类型推导**逆操作建议**：

- 对话响应 `ChatResponse` 新增 `rollbackPlan` 字段，确认执行后随结果一并返回；
- 对有幂等逆操作的命令给出补偿命令；对无幂等逆操作的命令（如 `systemctl restart`）给出明确的人工恢复指引，例如：
  > restart 无幂等逆操作，如需回滚请依据变更前状态手动恢复服务 nginx
- 全程 dry-run 安全，不会真正改动系统。

### 3.2 调用

```bash
curl -X POST http://localhost:8080/api/ops/rollback/[traceId]
```

返回该 traceId 对应变更的补偿回放结果与回滚指引。

## 4. 跨源根因分析（RCA · L1–L3）

### 4.1 设计

`CrossSourceRca` 将**主动巡检的指标发现**与**错误日志**做跨源关联，输出结构化根因洞察 `RcaInsight`，并给出整体严重级别 `overallLevel`：

| 级别 | 含义 |
| --- | --- |
| L0 | 无显著风险 |
| L1 | 单点告警，需关注 |
| L2 | 多源相关，需尽快处置 |
| L3 | 强关联高危，需立即升级人工 |

每条洞察包含：`level`（级别）、`correlation`（关联证据）、`rootCause`（根因判断）、`disposition`（处置建议）、`confidence`（置信度）。

#### 4.1.1 时间窗口跨源关联（非规则查表）

为避免「磁盘满→L3」式的单纯规则映射，`CrossSourceRca` 注入 `InspectionProperties`，按**时间窗口**将指标异常与日志事件做因果匹配，而非仅凭单项指标查表定级：

- 每类指标声明其**预期日志类别**，例如 `mem-usage → {OOM}`、`disk-usage → {DISK_FULL, IO}`；
- 仅当指标异常与对应类别的日志事件（来自巡检的 `recentLogEvents`，见 [11-主动巡检与风险预警.md](11-主动巡检与风险预警.md) §3.1）落在**同一时间窗口**（`ops.inspect.log-window-minutes`，默认 5 分钟）内，才判定为**同一根因**；
- 命中跨源关联时相应**抬升严重级别与置信度**（例如内存指标飙高且 5 分钟内出现 OOM 日志 → 关联为同一根因，升至 L2/L3）；未命中则保持单点级别，不虚高。

#### 4.1.2 LLM 根因总结（可选增强）

在配置了真实国产模型（`llm.isReal()`）时，`RcaLlmSummarizer` 会对关联结果做一次自然语言根因总结，给出比规则模板更贴合上下文的解释；**mock 模式或调用异常时自动回退为 `null`**，沿用规则化结论，从而保证 33 项评测与演示的字节级可复现。

### 4.2 调用

```bash
curl http://localhost:8080/api/ops/rca
```

返回 `Result<Map>`，含 `report`（巡检报告）与 `rca`（根因结果）两部分；并附带 `llmEnabled` 标志：当接入真实模型时为 `true`，额外返回 `llmSummary`（LLM 根因总结）与 `llmProvider`（模型提供方），mock 模式下为 `false` 且不含 summary。

### 4.3 示例（云服务器实测）

磁盘使用率 CRITICAL + 错误日志 CRITICAL 经时间窗口关联为同一根因：

- **Level**：L3
- **Root Cause**：磁盘空间接近写满，极可能导致服务写入失败、日志中断或数据库异常
- **Disposition**：立即升级人工并告警；按处置预案执行，全程保留一键回滚账本
- **Confidence**：90%
