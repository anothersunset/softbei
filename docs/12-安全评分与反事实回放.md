# 安全评分与反事实回放（差异化增强）

> 对标官方「多层级安全（静态风险评估 30% / 动态意图审计 35% / 受限执行 35%）」与「执行确定性」要求，把已有的三层护栏能力**量化、可视化**，并新增**反事实回放**让护栏的价值「看得见」。

## 一、为什么做这两个

官方高分攻略把**安全校验**列为功能完整性（≈60%）的核心得分项，并给出明确的三层权重；同时把「执行确定性」列为常见失分点。我们的护栏三层（静态规则 / 动态意图审计 / 最小权限执行）此前已实现，但缺两样东西：

1. **一个对齐官方权重的安全分** —— 让评委一眼看到「安全做到了什么程度」。
2. **一个让护栏价值可感知的瞬间** —— 当命令被拦截，回答「如果放行会发生什么」。

## 二、安全护栏综合评分 SecurityScore

按官方权重把单次请求的三层表现折算为 0–100 分：

| 维度 | 满分 | 评分依据（确定性） |
| --- | --- | --- |
| 静态风险评估 | 30 | 红线规则精准命中拦截=30；全只读安全=28；标记需确认=22 |
| 动态意图审计 | 35 | 注入被拦截=35；高危意图二次过滤=33；识别需确认=29；正常通过=32 |
| 受限执行 | 35 | 未落地任何变更=35；仅 dry-run/只读=33；受限账号下的授权真实变更=27 |

等级：A·稳健（≥90） / B·良好（75–89） / C·需关注（60–74） / D·高风险（<60）。

- 实现：`guard/SecurityScore.java`（记录）+ `guard/SecurityScorer.java`（纯计算）。
- 接入：`OpsAgentController` 在 `pipeline.chat()` 返回后基于响应数据计算并写入 `ChatResponse.securityScore`，**不参与 status 判定**。
- 每个维度附带 `notes` 文字说明，可直接用于处置报告与雷达图。

## 三、反事实回放 Counterfactual Replay

对每条被 **BLOCK / EXECUTABLE / IRREVERSIBLE** 的命令做**纯静态分析**（不执行任何命令），推演「若放行会发生什么」：

- **不可逆等级**：NONE / LOW / MEDIUM / HIGH / CATASTROPHIC
- **影响范围**：从命令中抽取的受影响路径/资源
- **最坏后果**：一句话描述
- **回滚建议**：对应的逆操作 / 恢复路径

示例（`rm -rf /`）：

| 字段 | 值 |
| --- | --- |
| 不可逆等级 | CATASTROPHIC |
| 影响范围 | 递归删除根目录下的全部文件 |
| 最坏后果 | 整个系统文件被清空，主机立即不可用 |
| 回滚建议 | 无法软回滚，只能依赖整机快照/备份重装恢复 |

- 实现：`guard/ImpactEstimate.java`（记录）+ `guard/CounterfactualAnalyzer.java`（启发式静态分析）。
- 接入：`OpsAgentController` 调用 `analyze(decisions)`，写入 `ChatResponse.counterfactual`。
- 覆盖模式：根删除 / 磁盘格式化覆写 / 递归删除 / 递归改权限 / 服务停机重启 / 网络与防火墙变更 / 数据库删除 / 软件包变更 / 文件改写；其余变更类降级为 LOW 并提示先行 dry-run。

## 四、对评分的价值

- **功能完整性（≈60%）**：把「安全校验」从隐式能力变成显式、可量化的安全分。
- **创新与实用性**：反事实回放是官方清单之外的差异化记忆点——把抽象「护栏」变成「它刚刚救了一次场」。
- **演示**：控制台可呈现安全分雷达 + 被拦命令的影响预估卡片（前端接入为后续步骤）。

## 五、不破坏现有能力

- 两项均为响应返回后的**纯加法计算**，不修改裁决、不修改 status。
- 反事实分析**只读、零副作用**，不执行任何命令。
- 评测 `ScenarioEvaluationTest` 仅断言 status，因此 **33/33 保持不变**。
