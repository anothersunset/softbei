# 六项问题整改闭环

> 目标：把“客观评价中指出的不足”逐项转成可运行、可测试、可答辩的证据。本文只记录当前代码已落地的能力，不把路线规划包装成已完成。

## 1. 测试数量多但深度存疑

整改：
- 新增 `OpsPipelineDeepIntegrationTest`，覆盖 EXECUTABLE/IRREVERSIBLE 首轮拦截、人工确认后复用同一缓存计划、dry-run 执行、回滚账本、伪造 trace 不能绕过确认门。
- 新增 `PromptInjectionNormalizationTest`，覆盖零宽字符、`\xNN` 编码、语义绕过、新型反弹 shell 工具变体。
- 新增 `CrossSourceRcaDepthTest`，覆盖 DISK_FULL、DEPENDENCY/NETWORK、CONFIG 三类跨源关联深度。

验证：
- `mvn -q -Dtest=PromptInjectionNormalizationTest,OpsPipelineDeepIntegrationTest,CrossSourceRcaDepthTest test`

## 2. 没有前端

整改：
- `backend/src/main/resources/static/index.html` 已从空壳升级为可演示控制台。
- 前端覆盖自然语言指令、九阶段闭环、任务级执行计划、结果/链路/报告/巡检/RCA/红蓝对抗/JSON/SSE 实时视图。
- 顶部 runtime badge 会从 `/api/ops/runtime` 读取当前 `MOCK/REAL`、provider、dry-run/live-exec 状态。

验证：
- 启动后访问 `http://127.0.0.1:8080/`。
- 页面应显示服务健康、运行模式、工具列表，并可完成 READONLY/EXECUTABLE/IRREVERSIBLE/BLOCK 四类风险演示。

## 3. 护栏核心偏规则层，LLM 深度不足

整改：
- 入口护栏增加 Unicode NFKC 归一化、控制/零宽字符清理、`\xNN` 解码，减少编码绕过。
- `risk-rules.yaml` 扩展 override/bypass/disable validation/safeguards/checks/controls 等语义表达。
- 运行态接口明确边界：`LLM proposes; deterministic guards decide execution`。即 LLM 负责生成方案，执行权由确定性护栏、最小权限执行器、人工确认门共同控制。

仍需如实说明：
- 当前主执行判定仍以确定性规则为安全基座，这是工程安全选择，不应伪装成纯 LLM 语义判定。
- 真实模型用于推理和 RCA 总结增强；是否放行命令不依赖模型自我声明。

## 4. RCA 跨源关联较浅

整改：
- RCA 测试覆盖从 OOM/DISK_FULL/IO 扩展到 DISK_FULL、DEPENDENCY、NETWORK、CONFIG 的组合场景。
- `CrossSourceRcaDepthTest` 验证 L3 条件必须来自指标异常与日志事件在时间窗口内共同出现，CONFIG 单日志漂移保持 L2，避免“一条日志直接升 L3”。

仍需如实说明：
- 当前是轻量跨源关联，不是完整 APM 拓扑推断。跨服务依赖图、配置版本 diff、网络分区抓包证据仍属于后续增强。

## 5. 真实 LLM 与 Mock 边界模糊

整改：
- 新增 `/api/ops/runtime`，返回 `llmProvider`、`llmMode`、`llmReal`、`mockDeterministic`、`dryRun`、`maxStepsPerRequest`。
- 前端顶部直接展示当前模式，例如 `mock(mock) · dry-run` 或 `real(xiaomi) · live-exec`。
- 集成测试验证默认评测环境会明确暴露 `MOCK + dryRun=true`。

答辩口径：
- Mock 路径用于离线可复现评测。
- 真实 LLM 路径用于展示自然语言推理与 RCA 总结质量。
- 两者共用同一护栏、确认门、最小权限执行器和审计链路。

## 6. 单人项目缺乏多视角质疑

整改：
- 新增 PR 模板，要求每次变更填写风险面、测试证据、Mock/真实 LLM 边界、前端验证、回滚方案。
- 六项整改文档把“已完成”和“仍需如实说明”分开，避免自我评价过度乐观。

仍需如实说明：
- 文档和模板不能替代真正的多人 code review。比赛提交前建议至少找 1 名同学按 PR 模板做一次外部审阅，并把审阅记录追加到 `_evidence/`。
