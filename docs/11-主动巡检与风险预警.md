# 主动巡检与风险预警

> 对标官方高分攻略中「主动巡检 vs 被动响应」与「多维感知成熟度（预测性感知）」的要求，本模块在原有「感知-决策-护栏」闭环之外，增加一条**主动发起、只读、可溯源**的健康巡检通道，将运维从「出事后问」提前到「隐患早发现」。

## 1. 为什么选这个能力

官方评分中**功能完整性约占 60%（第一大得分项）**，核心在于 **MCP 工具丰富度 + 安全校验 + 推理可追溯**。主动巡检同时命中三个加分点：

| 维度 | 如何命中 |
| --- | --- |
| MCP 工具丰富度 | 新增 `health_inspect` 工具，自动出现在 MCP `tools/list` 与控制台插件列表 |
| 创新与实用性 | 从被动响应进化为主动巡检 + 风险预警（预测性感知） |
| 文档与演示 | 控制台新增「主动巡检」标签页，一键输出健康评分与预警清单 |

关键是：该能力**全程只读**，不触发任何变更，不依赖 root，也不影响护栏/评测体系的确定性。

## 2. 设计架构

```
HealthInspectTool (MCP 插件, implements ActiveTool)
        │  /api/ops/inspect (REST)        │  InspectionScheduler (定时, 默认关)
        └─────────────────┐  ┌───────────────────┘
                          ▼  ▼
                    InspectionService.inspect()
                          │  通过 LeastPrivilegeExecutor.runReadOnly() 采集
          ┌──────┬──────┬──────┼──────┬──────┬──────┐
          磁盘   内存   负载   僵尸进程  监听端口  错误日志
          df -P  free  uptime  ps      ss        journalctl
                          │
                    InspectionReport { healthScore, overall, findings[], sources[], traceId }
```

- **复用最小权限执行器**：所有采集走 `runReadOnly()`，与主护栏同一条受限执行通道；单条采集失败只记为 `UNKNOWN`，不报错、不中断整体巡检。
- **独立 trace**：巡检自带一条追溯链（`audit.newTrace` → 阶段 `INSPECT` → `complete`），与对话主管道互不干扰。

### 2.1 ActiveTool 标记接口与感知随动隔离

由于感知阶段会遍历所有 `AgentTool` Bean，若 `HealthInspectTool` 被当作普通感知工具，则每次对话都会误跑一次全量巡检。为此引入标记接口：

```java
public interface ActiveTool extends AgentTool {}
```

- `SenseNode` 遍历时 `if (tool instanceof ActiveTool) continue;` — 主动巡检工具**不参与被动感知**；
- 但 `McpController` / `OpsAgentController` 仍将其暴露到 MCP 工具列表与 `/api/ops/tools`，丰富度不受损。

## 3. 健康评分与分级

每个检查项输出一个 `InspectionFinding`，分级为 `OK / WARN / CRITICAL / UNKNOWN`。总分从 100 起扣：

| 严重程度 | 扣分 |
| --- | --- |
| CRITICAL | -25 |
| WARN | -10 |
| UNKNOWN | -3 |

最终 `healthScore` 下限为 0；`overall` 由最严重项决定（含 CRITICAL → `CRITICAL`，含 WARN → `WARNING`，否则 `HEALTHY`）。

### 阈值（外置于 application.yml · ops.inspect.*）

| 检查项 | 采集命令 | WARN | CRITICAL |
| --- | --- | --- | --- |
| 磁盘使用率 | `df -P` | 80% | 90% |
| 内存使用率 | `free` | 85% | 95% |
| 每核负载 | `uptime` + `nproc` | 1.0 | 2.0 |
| 僵尸进程数 | `ps -eo stat=` | 1 | 5 |
| 错误日志条数 | `journalctl -p 3 -n 200` | 20 | 100 |
| 监听端口 | `ss -H -tuln` | — | —（仅供参考） |

> 阈值可按环境在 `application.yml` 中调整，无需改代码。

### 3.1 结构化日志事件采集（供跨源关联）

除「错误日志条数」这一聚合阈值外，巡检还会通过 `journalctl -p 3 -n 200 -o short-iso --no-pager` 采集近期错误日志，并解析为带时间戳的结构化事件 `LogEvent{time, epochMillis, kind, message}`，随 `InspectionReport.recentLogEvents` 返回；采集失败（退出码非 0）时优雅降级为空，不影响巡检主流程。

- `kind` 由日志内容归类，如 `OOM`（内存耗尽）、`DISK_FULL`（磁盘写满）、`IO`（I/O 错误）等；
- 这些带时间戳的事件是「**指标↔日志时间窗口关联**」的输入，关联时间窗口由 `ops.inspect.log-window-minutes`（默认 5 分钟）控制；
- 跨源根因分析如何消费这些事件（同窗口命中即关联为同一根因、抬升级别与置信度），详见 [14-执行兜底与跨源根因分析.md](14-执行兜底与跨源根因分析.md) §4。

## 4. 调用方式

### REST

```bash
curl -X POST http://localhost:8080/api/ops/inspect
# 或 GET
curl http://localhost:8080/api/ops/inspect
```

返回 `Result<InspectionReport>`，`data` 结构：

```json
{
  "inspectId": "...",
  "traceId": "...",
  "timestamp": "...",
  "healthScore": 82,
  "overall": "WARNING",
  "summary": "共 6 项检查，1 项告警，0 项严重",
  "findings": [
    { "id": "disk", "category": "磁盘", "severity": "WARN", "title": "根分区使用率偏高",
      "metric": "disk.usage", "observed": "83%", "threshold": "WARN>=80% CRITICAL>=90%",
      "evidence": "/dev/vda1 ... 83% /", "suggestion": "清理日志或扩容" }
  ],
  "sources": ["df -P", "free", "uptime", "ps", "ss", "journalctl"],
  "elapsedMs": 42
}
```

### MCP

工具名 `health_inspect`，无参，通过 `POST /mcp/rpc` 的 `tools/call` 调用。

### 定时自动巡检

默认关闭。设 `ops.inspect.scheduled-enabled=true`（或环境变量 `OPS_INSPECT_SCHEDULED=true`）后，`InspectionScheduler` 按 `interval-ms`（默认 5 分钟）周期性执行并记入追溯。

## 5. 控制台演示

右侧面板新增「**主动巡检**」标签页：点击「立即巡检」后呈现健康评分仪表、总体状态、逐项风险预警（按 OK/WARN/CRITICAL 色彩区分）与只读数据源清单。

## 6. 与推理决策选型的关系（选型依据）

官方推荐「**Plan-Execute 全局 + 子任务 ReAct**」的混合范式：主护栏闭环的「推理→校验→执行→分析」即全局 Plan-Execute；本模块的单项检查（采集→比对阈值→得出 finding）是轻量的观察-判断循环，与 ReAct 子任务理念一致。巡检结果也可作为后续对话的「预测性感知」输入，为根因分析的「指标-日志跨源关联」提供依据。
