# 已知限制与演进路线（Known Limitations & Roadmap）

> 诚实声明边界比掩盖问题更能赢得评委信任。本文集中说明本作品当前的能力边界与后续规划。

## 1. 已知限制

### 1.1 护栏检测为规则层
- 当前注入/风险识别以 `risk-rules.yaml` 的模式匹配为主，优点是**确定、可复现、可审计**；局限是对全新变体的泛化依赖规则覆盖面。
- 缓解：已引入输入规范化、盲测集（`blindset-corpus.yaml`）与参数/子命令级命令面校验；例如 `find -delete/-exec` 会被 BLOCK，而普通 `find` 查询仍保持 SAFE，`docker/kubectl/crontab/sed/tar/rsync` 按只读/变更子命令分级。
- 边界：命令字符串匹配仍是第一道防线，不承诺穷举所有管道、脚本、同义工具绕过。答辩时应明确：预防性兜底依赖最小权限账号、sudoers/ACL/只读挂载等 OS 层控制；auditd 与 OpsAuditService 负责事后取证，不把审计说成拦截。

### 1.2 RCA 深度
- 已落地三类故障（OOM / DISK_FULL / IO）且主要在 L2 分级验证；跨服务依赖雪崩、网络分区、配置漂移等场景尚未覆盖。
- 路线：见 `docs/19-RCA故障分级矩阵.md` 中标记为「规划」的条目。

### 1.3 演示与可观测
- 当前演示为 asciinema 终端录屏，叙事张力有限；路线上补一段 90s「拦截→定位→回滚」叙事型成片（见 `docs/20-演示脚本与分镜.md`）。

### 1.4 工程成熟度
- 单人开发，覆盖面广。为避免「广而不深」，已将**安全护栏闭环**（意图分级+反事实回放+回滚账本）作为招牌深度线，其余能力服务于这条主线。

### 1.5 入口信任边界
- 默认服务绑定 `127.0.0.1`，适用于本机演示；内网/公网部署需要显式设置 `OPS_BIND_ADDRESS=0.0.0.0`。
- REST `/api/**` 与 HTTP MCP `/mcp/**` 支持同一 `OPS_API_TOKEN` 入口令牌；绑定到非回环地址但未配置 token 时，服务会拒绝启动。
- stdio MCP 属本机进程间通道，不通过 HTTP token 鉴权，安全边界在启动该进程的本机用户权限。
