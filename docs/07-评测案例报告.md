# 智御 · OS 智能运维 Agent — 安全护栏评测案例报告

## 1. 评测目标

不以“自说自话”证明安全能力，而是用一套**可复现、可量化**的场景语料 + 自动回放，给出安全护栏的客观指标（借鉴智迁云枢“有 case、有对照实验”的思路）。

## 2. 评测方法

- **语料**：`backend/src/main/resources/eval/scenarios.yaml`，共 **33 条**场景，覆盖体检/网络/日志/磁盘/进程/变更/危险删除/元字符注入/提示词注入。
- **驱动**：`ScenarioEvaluationTest`（`@SpringBootTest`）逐条调用 `OpsPipeline.chat()`，校对最终 `status`；EXECUTABLE/IRREVERSIBLE 用例追加一次 `confirm=true` 验证二次确认闭环。
- **确定性**：测试环境强制 `provider=mock`、`dry-run=true`（见 `src/test/resources/application.yml`），期望结果由 Mock 推理路由 + 护栏规则推导，**100% 可复现**。

### 运行方式

```bash
cd backend
mvn -Dtest=ScenarioEvaluationTest test
```
控制台会打印「安全护栏评测结果」：总通过率、分类通过率、以及三项安全关键指标。

## 3. 语料构成

| 分类 | 数量 | 期望状态 |
| --- | --- | --- |
| 系统体检 / 网络排查 / 日志排查 | 8 | EXECUTED（只读 READONLY） |
| 磁盘清理 / 进程处置 / 服务处置 / 文件变更 | 9 | REVIEW_PENDING -> 确认后 EXECUTED |
| 危险删除 / 危险命令 | 7 | BLOCKED |
| 元字符注入 | 2 | BLOCKED |
| 提示词注入 | 7 | INJECTION_BLOCKED |
| **合计** | **33** | — |

## 4. 预期安全关键指标

> 以下为确定性环境下的预期值，以实际 `mvn test` 控制台输出为准。

| 指标 | 预期 |
| --- | --- |
| 总通过率 | 33/33 = 100% |
| 危险命令拦截率（BLOCKED） | 9/9 = 100% |
| 提示词注入识别率 | 7/7 = 100% |
| 正常指令误拦率（误拦/误判注入） | 0/17 = 0% |
| EXECUTABLE/IRREVERSIBLE 二次确认闭环 | 9/9 = 100% |

说明：“误拦率 0”表明护栏在拦住全部危险/注入的同时，**不会误伤正常运维指令**，兼顾了安全性与可用性。

## 5. 设计说明：为何校验作用于“命令”而非“意图”

部分模糊自然语言（如“删除根目录”）不一定生成真正的危险命令，此时无命令可拦；而一旦生成的具体命令命中红线/关键路径/元字符，护栏零放过。语料中的危险用例均以**明确可执行的危险命令/路径**构造，确保拦截可复现，这也体现了“命令层二次过滤”的纵深防御设计。

## 6. MCP 协议合规验证

MCP（Model Context Protocol）是 Agent 与 OS 运维能力之间的桥梁，其协议合规性直接决定“运维插件化”的互操作性。本节针对 `/mcp/rpc`（HTTP transport）做了一组独立的 JSON-RPC 2.0 + MCP 规范一致性验证。

- **被测对象**：`McpController` / `McpDispatcher` / `McpToolSpec`（设计说明见 `docs/02-功能设计.md` §3.8）。
- **驱动脚本**：`scripts/verify_mcp.sh`，对运行中的服务依次发起 8 组用例（MCP-01~08）。
- **执行环境**：Ubuntu 22.04（2 核 / 1.6 GiB 可用）、端口 8080、`provider=xiaomi (mimo-v2.5-pro)`、`dry-run=true`。
- **原始证据**：完整请求/响应见 `docs/MCP协议合规验证报告.md`。

### 6.2 关键合规点

- **版本协商**：`initialize` 回显客户端协议版本（服务端支持 2024-11-05 / 2025-03-26 / 2025-06-18），并返回 `serverInfo` 与 `instructions`。
- **通知语义**：`notifications/initialized` 无 `id`，服务端返回 HTTP 202 且响应体为空，符合 JSON-RPC 通知规范。
- **工具注解收敛**：6 个运维工具均声明 `annotations.readOnlyHint=true` 且 `inputSchema.additionalProperties=false`，从协议层约束“幻觉参数”，与安全护栏的纵深防御一脉相承。
- **标准错误码**：未知工具 `-32602`、未知方法 `-32601`、非法请求 `-32600`，错误处理路径完整。

### 6.3 插件化传输层（stdio transport）

除 HTTP transport 外，项目额外提供 **stdio transport** 的 MCP Server（`McpStdioServer`，通过 `--mcp-stdio` 启动），与 HTTP 共享同一套 `McpDispatcher` 核心逻辑，可作为独立巡检插件进程经 stdin/stdout 接入主 Agent，直接呼应赛题“MCP 运维插件化”的要求。演示脚本见 `scripts/mcp-stdio-demo.sh`，可观察 `initialize`/`tools/list`/`tools/call` 经 stdio 的完整往返，以及通知无响应的语义。

## 7. 结论

通过可复现的 33 条场景语料与自动回放，量化证明了安全护栏在危险命令拦截、提示词注入识别、变更二次确认上的有效性，且对正常运维指令零误拦；叠加 MCP 协议合规验证 15/15 全部通过、以及 HTTP + stdio 双传输的插件化能力，为评审提供了一条从安全护栏到协议互操作的完整证据链。
