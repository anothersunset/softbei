# 运维知识检索层 ContextRetriever 设计说明

## 1. 背景与价值

原有闭环中，Agent 能感知系统状态、能让 LLM 生成处置方案，但方案缺少「依据」。本层引入轻量离线检索器，在推理前为模型注入可引用、可溯源的运维知识，使项目从「会操作」升级为「有证据地操作」，同时增强答辩时的可信度。

## 2. 架构位置

RETRIEVE 阶段位于 SENSE 与 REASON 之间，构成八阶段闭环：

```
RECEIVE -> INJECTION_GUARD -> SENSE -> RETRIEVE -> REASON -> GUARD -> EXECUTE -> ANALYZE
```

- 节点类：`OpsPipeline.RetrieveNode`，stage=`RETRIEVE`，agent=`ContextRetriever`。
- 输出：`{count, citations:[{title,source,kind,score,snippet}], degraded}`，同时将 `List<Evidence>` 写入上下文 `retrieval`，随 `ChatResponse.retrieval` 返回前端。

## 3. 知识来源（三路融合）

| 来源 | kind | 内容 |
| --- | --- | --- |
| `classpath:kb/*.md` | doc | 麒麟 V11 / LoongArch 文档、磁盘/CPU/端口/日志 runbook、危险命令说明（按 `## ` 二级标题切块）|
| `risk-rules.yaml` | rule | 高危阻断规则、关键路径保护、变更类/只读命令清单（由 `RiskRuleLoader` 加载）|
| 近期 trace | history | `OpsAuditService.recent(20)` 中的同类历史指令与处置结论（排除当前 trace）|

知识库文档为纯静态 Markdown，新增/修改无需改代码，便于评审与扩充。

## 4. 检索算法

- **分词**：英文/数字按词（长度 ≥ 2），中文按相邻二元组（bigram，码点 0x4E00–0x9FFF）。
- **打分**：BM25-lite——对查询中出现于文档的每个词累加 `idf = log(1 + N/df)`，`N` 为语料文档数、`df` 为含该词文档数。
- **返回**：得分 > 0 者按降序取 topK（默认 4），snippet 为压缩空白后的首段 ≤ 180 字。
- **健壮性**：任何异常（加载失败/离线/空库）均返回空列表，绝不抛出，保障主链路可用。

## 5. 确定性与安全

- 检索到的依据**仅在 `llm.isReal()` 为真时**追加到提示词的 `[运维知识/依据]` 段；
- mock 模型的提示词保持字节级不变，因此 33 项确定性评测（见 docs/07）不受影响，可复现；
- 检索不会绕过任何安全护栏：RETRIEVE 仅提供参考，后续 GUARD 仍对每条命令独立二次裁决。

## 6. 前端展示

- 决策结果页新增「知识依据 · 引用溯源」面板，按 doc/rule/history 区分徐色展示标题、来源与摘要；
- 推理链路溯源新增 RETRIEVE 阶段节点；
- 处置报告「模型计划」小节附「参考依据」列表，导出 Markdown / 打印 PDF 同步携带。

## 7. 可扩展性

当前为轻量 BM25-lite，接口（`retrieve(query, topK, excludeTraceId)` 返回 `List<Evidence>`）保持稳定。后续可在不改上层的前提下，平滑升级为向量检索（如 BGE-M3 + 向量库）或混合检索，复用原项目 CRAG/RRF 思路。
